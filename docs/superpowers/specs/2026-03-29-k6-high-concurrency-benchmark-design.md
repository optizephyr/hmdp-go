# k6 高并发压测设计

## 背景

项目当前的核心价值点是高并发读写链路：商户详情读取依赖缓存，秒杀链路依赖 Redis Lua 预检、RocketMQ 异步削峰和数据库事务兜底。

用户希望使用 k6 对四类场景做探索，但在梳理项目目标后，最终收敛为两个主测试：纯读链路高并发测试与极限秒杀瞬时高并发测试。

## 目标

- 证明系统能扛住高并发流量。
- 用两条主线讲清楚系统能力：读链路性能与秒杀链路峰值承载。
- 在秒杀场景中同时验证业务正确性：不超卖、一人一单。
- 让压测结论适合用于项目介绍、README 补充和面试表达。

## 决策结论

- 只保留两个主测试场景。
- 不把恶意流量穿透测试作为主测试，因为后续计划引入 Redis 布隆过滤器，单独压穿透的叙事价值下降。
- 不把全链路混合压测作为主测试，因为它更偏验收型，结果解释成本更高，容易稀释主线叙事。

## 范围

### 场景 1：纯读链路高并发测试

- 接口：`GET /shop/:id`
- 流量入口：直连 Go 后端，端口 `8081`
- 请求模型：随机命中一批存在的商户 ID
- 压测形态：`1 分钟预热 + 5 分钟稳态压测`
- 并发模型：从 `50 VUs` 逐步升到 `200 VUs`
- 目标：证明缓存命中后的读链路能稳定承受高并发访问，并覆盖冷启动后的热态表现

### 场景 2：极限秒杀瞬时高并发测试

- 接口：`POST /voucher-order/seckill/:id`
- 流量入口：直连 Go 后端，端口 `8081`
- 请求模型：随机账号池 + 固定秒杀券 ID
- 库存基线：使用独立测试券，每轮压测前将该券库存重置为固定值，并清空该券已有订单记录，保证每次结果可比
- 压测形态：`30 秒预热 + 1 分钟瞬时冲顶 + 2 分钟回落观察`
- 并发模型：从 `50 VUs` 逐步升到 `300 VUs`，冲顶阶段维持满载
- 目标：证明秒杀链路在高并发下既能保持吞吐，又能保证业务正确性，并观察峰值后的排队回落

## 业务正确性验证

秒杀场景除了压性能，还必须验证以下约束：

- 成功订单数不超过库存。
- 同一用户最多成功下单一次。
- 订单表中不出现重复的 `user_id + voucher_id` 组合。
- 库存不会被扣成负数。
- 以上结论以数据库最终状态为准，必须在压测结束后等待 RocketMQ 消费积压清空后再核验。
- 若在 `2 分钟` 内积压未归零，则本轮秒杀测试判定失败。

## 指标

### 读链路指标

- `QPS`
- `p95`
- `p99`
- 错误率
- 响应时间分布
- 通过率最低不得低于 `99%`
- `p95` 不得高于 `200ms`

### 秒杀链路指标

- `QPS`
- `p95`
- `p99`
- 接口成功率
- 订单成功数
- 重复订单数
- 库存结果是否异常
- RocketMQ 是否出现明显堆积
- 系统错误率（超时、5xx、异常返回）不得高于 `1%`
- 业务拒绝单独统计，不计入系统错误率
- RocketMQ 消费积压在压测结束后 `2 分钟` 内必须归零，否则判失败
- 最终订单数不得超过库存，重复订单数必须为 `0`
- 库存最终值不得小于 `0`

## 执行方式

- 使用 k6 编写两组场景脚本，或在同一脚本中按场景切换。
- 读取一批已存在的商户 ID 作为读链路数据源。
- 读取一批已登录用户的 token 作为秒杀用户池；token 生成方式可复用仓库里现有的 Redis 写入思路。
- 秒杀券 ID 固定为一个已配置的秒杀券，避免请求目标漂移。
- 压测时直接打 Go 服务，不经过 Nginx，减少网关层变量对结论的干扰。
- 每次执行都记录压测开始时刻、峰值并发、持续时长、实际完成请求数，便于复盘对比。
- 每次执行前先重置秒杀券库存和该券关联订单，确保测试基线一致。

## 验收口径

- 读链路在高并发下保持稳定，无明显错误率飙升。
- 秒杀链路在高并发下保持稳定返回。
- 秒杀结果同时满足“不超卖”和“一人一单”。
- 压测结果可以直接用于项目说明和面试讲述。

## 非目标

- 不做穿透压测。
- 不做全链路混合压测。
- 不在本轮压测中验证布隆过滤器效果。
- 不将 Nginx 网关承压作为本轮主结论。
